# Zerone Online Judge 요구사항 정리

## 1. 문서 목적

이 문서는 새로 만들 `Zerone Online Judge`의 요구사항을 정리하기 위한 작업 문서다.
구현 전에 제품 범위, 권한 구조, 화면 흐름, 백엔드 책임, 채점 인프라 요구사항을 합의하는 것이 목적이다.

이 문서는 논의하면서 계속 갱신한다.

## 2. 프로젝트 전제

- `hs_judge`는 보관용 레거시이며, 신규 제품의 디자인/정보 구조/사용자 흐름 설계 기준으로 삼지 않는다.
- 레거시 코드는 수정하지 않는다.
- 실제 신규 개발은 현재 루트의 `frontend`, `backend` 기준으로 진행한다.
- 현재 단계에서는 `대회 규정/운영 정책 정리 -> DDD 정리 -> 구현` 순서로 진행한다.
- 지금 단계에서는 코드를 작성하기보다 요구사항, 규정, 도메인 모델을 먼저 정리한다.

## 3. 신규 프로젝트 방향

이 프로젝트는 일회성 대회 사이트가 아니라 계속 사용할 수 있는 온라인 저지 플랫폼을 목표로 한다.
성격은 `백준의 대회 기능을 참고한 자체 코딩테스트/대회 플랫폼`에 가깝다.

핵심 방향:

- 레거시를 답습하지 않고 디자인과 사용자 흐름을 처음부터 새로 설계한다.
- 대회 규정, 참가 정책, 운영 정책을 먼저 정리한다.
- 확정된 규정과 정책을 기준으로 DDD의 용어, 경계, 책임을 정리한다.
- 프론트는 새로 React 기반으로 구현한다.
- 백엔드는 FastAPI 기반으로 새로 구현한다.
- 채점기는 백엔드와 분리된 독립 시스템으로 운영한다.
- 대회 생성, 예약 오픈, 문제 관리, 참가 팀 관리, 채점 확장성을 처음부터 고려한다.
- 가장 중요한 비기능 요구사항은 `안정성`, `대규모 트래픽 대응`, `로드 밸런싱`, `보안`이다.

## 4. 기술 스택 확정 사항

### 프론트엔드

- React + TypeScript + Tailwind CSS 기반으로 개발한다.
- 프론트 디자인과 주요 웹 구조는 신규 요구사항과 DDD 결과를 기준으로 새로 설계한다.

### 백엔드

- Python FastAPI 기반으로 개발한다.
- 백엔드는 확정된 규정, 정책, DDD 모델을 기준으로 구현한다.
- 백엔드는 `공개 웹/API 서버`와 `비공개 채점 서버 연동 영역`을 중심으로 책임을 나눈다.
- 외부 사용자가 직접 접속하는 대상은 웹/API 서버뿐이어야 한다.

### 채점 시스템

- 백엔드와 별도 프로세스/서비스로 분리한다.
- 제출은 FIFO 기준으로 처리한다.
- 대량 트래픽 상황을 전제로 로드 밸런싱 구조를 포함해야 한다.
- 채점 실행은 반드시 Docker 샌드박스에서 수행한다.
- 채점 서버는 공개 인터넷에 직접 노출하지 않는다.
- 웹/API 서버와 채점 서버 간 통신은 Tailscale 사설망을 통해 수행하는 방향으로 설계한다.

## 5. 설계 진행 순서

현재 기준 설계 순서는 아래와 같다.

1. 대회 규정, 참가 정책, 공개 정책, 스코어보드 정책, 재채점 정책을 먼저 정리한다.
2. 팀 참가 모델을 정리한다.
3. 위 정책을 기준으로 DDD의 bounded context, aggregate, 용어를 정리한다.
4. 그 결과를 바탕으로 디자인, 정보 구조, 화면 흐름을 새로 설계한다.
5. 문서 합의가 끝난 뒤 `frontend`, `backend`, 채점 시스템 구현에 들어간다.

중요 원칙:

- 레거시 화면 구조나 사용자 흐름은 기준으로 삼지 않는다.
- 초기 설계는 문서와 규정에서 출발한다.
- DDD는 구현 직전에 덧붙이는 단계가 아니라, 구현 전에 도메인 규칙을 정리하는 단계로 다룬다.

## 6. 제품 목표

이 시스템은 아래를 만족해야 한다.

- 여러 대회를 지속적으로 개최할 수 있어야 한다.
- 특정 권한 사용자가 대회를 생성하고 예약 공개할 수 있어야 한다.
- 대회별 운영자가 문제와 참가 팀을 관리할 수 있어야 한다.
- 참가 팀은 대회 기간 중 문제를 조회하고 제출할 수 있어야 한다.
- 제출은 안정적으로 큐잉되고 채점되어야 한다.
- 트래픽이 몰릴 때도 서비스와 채점이 안정적으로 유지되어야 한다.

## 7. 사용자 역할

현재 기준 역할은 아래처럼 정의한다.

### 서비스 마스터

- 시스템 전반을 관리하는 상위 권한
- 이 서비스의 총괄 책임자다.
- 서비스 내 모든 기능과 리소스를 관리할 수 있다.
- 서비스 마스터만 대회를 개최할 수 있다.
- 대회를 생성하고 오픈할 수 있다.
- 대회 예약 오픈 설정 가능
- 대회별 운영자를 초대할 수 있다.
- 모든 대회의 설정, 운영자, 매니저, 참가 팀, 문제, 리소스, 제출, 채점 관련 설정을 관리할 수 있다.
- 서비스 매니저를 추가할 수 있다.
- 서비스 매니저 관리 페이지에서 서비스 매니저 추가/수정/삭제 및 권한 변경이 가능해야 한다.
- 필요 시 채점 서버 설정 변경 가능

### 서비스 매니저

- 서비스 마스터가 추가할 수 있는 전역 권한이다.
- 서비스 운영을 보조하는 상위 운영 권한이다.
- 구체적인 권한은 서비스 마스터가 설정할 수 있어야 한다.
- 서비스 매니저 권한은 고정 역할이 아니라 `권한 묶음` 기반으로 부여할 수 있어야 한다.
- 서비스 매니저 권한은 `전역 범위` 또는 `특정 대회 범위`로 부여할 수 있어야 한다.
- 서비스 매니저 관리 페이지에서 권한 변경 대상이 될 수 있어야 한다.
- 서비스 마스터에 의해 계정이 생성 또는 초대되어야 한다.
- 로그인은 이메일과 비밀번호 기반으로 수행한다.
- 세부 권한은 아래 권한 카탈로그를 기준으로 선택 부여할 수 있어야 한다.

### 대회 운영자

- 서비스 마스터/서비스 매니저와는 다른 대회 단위 권한이다.
- 특정 대회에 소속되어 운영 업무를 수행한다.
- 한 사용자가 여러 대회의 운영자가 될 수 있다.
- 권한 있는 사용자가 추가하면 이메일로 비밀번호 설정 메일을 받아 로그인할 수 있어야 한다.
- 해당 대회에 대한 모든 설정과 리소스를 관리할 수 있는 사람이다.
- 해당 대회의 대회 매니저를 초대할 수 있다.
- 해당 대회의 설정 페이지에서 대회 매니저의 권한과 추방 등을 관리할 수 있다.
- 해당 대회의 문제 추가/수정/삭제 가능
- 해당 대회의 문제 순서 정리 가능
- 해당 대회의 문제 리소스 관리 가능
- 해당 대회의 참가 팀, 문제, 스코어보드, 설정 등 전반을 관리할 수 있어야 한다.

### 대회 매니저

- 대회 운영자가 초대할 수 있는 대회 단위 권한이다.
- 특정 대회에 소속되어 운영 보조 업무를 수행한다.
- 대회 운영자 또는 관련 권한이 있는 사용자가 임명할 수 있어야 한다.
- 세부 권한은 해당 대회 설정 페이지에서 관리할 수 있어야 한다.
- 대회 운영자에 의해 추방될 수 있어야 한다.
- 한 사용자가 여러 대회의 대회 매니저가 될 수 있다.
- 권한 있는 사용자가 추가하면 이메일로 비밀번호 설정 메일을 받아 로그인할 수 있어야 한다.

### 참가 팀

- 대회 참가 단위는 `팀`이다.
- 모든 참가는 팀 기준으로 처리한다.
- 개인 참가도 `1인 1팀`으로 취급한다.
- 각 대회는 `1인 1팀`, `n인 1팀`, `팀원별 동시 접속 허용 여부`, `동시 접속 가능 수`를 참가 정책으로 가질 수 있어야 한다.
- 일부 대회는 `3인 1팀`으로 운영될 수 있으며, 이 경우 팀원 3명 모두 동시에 접속해 문제를 풀 수 있어야 한다.
- 팀 로그인 식별자는 대표자 1개로 고정하지 않고, 대회 정책에 따라 대표자 1개 또는 팀원별 여러 개를 등록할 수 있어야 한다.
- 대표 연락용 이메일은 별도로 관리할 수 있어야 한다.
- 팀명과 로그인 식별자들은 사전에 연결되어 저장되어야 한다.
- 팀은 참가 가능한 대회 조회
- 팀은 문제 조회
- 팀은 코드 제출
- 팀은 제출 현황 조회
- 팀은 결과 확인

### 역할 충돌 규칙

- 한 사용자는 여러 대회에서 서로 다른 역할을 가질 수 있다.
- 한 사용자가 어떤 대회에서는 운영자이고, 다른 대회에서는 팀 계정의 담당자일 수 있다.
- 한 사용자가 어떤 대회에서는 매니저이고, 다른 대회에서는 팀 계정의 담당자일 수 있다.
- 전역 역할인 서비스 마스터/서비스 매니저와 대회 단위 역할은 함께 가질 수 있다.
- 단, 같은 대회에서는 대회 운영자와 대회 매니저가 참가 팀으로 등록될 수 없다.
- 즉, 해당 대회의 운영자 또는 매니저는 그 대회에 참가할 수 없다.

### 대회 매니저 권한 카탈로그 초안

대회 매니저는 고정 권한 역할이 아니라, 해당 대회에 한정된 permission 묶음으로 권한을 부여받아야 한다.
대회 운영자 또는 관련 권한이 있는 사용자는 아래 권한을 개별적으로 부여할 수 있어야 한다.

`대회 기본 정보 권한`

- `contest.info.view`: 대회 개요 및 기본 정보 조회 가능
- `contest.info.update_overview`: 대회 개요 수정 가능
- `contest.info.update_organization`: 개최 기관 및 주최 단체 정보 수정 가능
- `contest.info.update_rule`: 대회 규정 및 안내 수정 가능
- `contest.info.update_schedule`: 대회 일정 수정 가능

`대회 참가 설정 권한`

- `contest.participant_setting.view`: 참가 관련 설정 조회 가능
- `contest.participant_setting.update`: 참가 팀 추가 가능 여부 등 참가 정책 수정 가능

`참가 팀 관리 권한`

- `contest.participant.view`: 참가 팀 목록 조회 가능
- `contest.participant.create`: 참가 팀 추가 가능
- `contest.participant.update`: 참가 팀 수정 가능
- `contest.participant.remove`: 참가 팀 삭제 가능

`문제 조회/관리 권한`

- `contest.problem.view`: 대회 문제 조회 가능
- `contest.problem.create`: 문제 추가 가능
- `contest.problem.update`: 문제 수정 가능
- `contest.problem.delete`: 문제 삭제 가능
- `contest.problem.reorder`: 문제 순서 변경 가능

`문제 리소스 및 채점 데이터 권한`

- `contest.problem.resource.view`: 문제 리소스 조회 가능
- `contest.problem.resource.manage`: 문제 리소스 관리 가능
- `contest.testcase.view`: 테스트케이스 조회 가능
- `contest.testcase.manage`: 테스트케이스 관리 가능
- `contest.generator.view`: 테스트케이스 제너레이터 조회 가능
- `contest.generator.manage`: 테스트케이스 제너레이터 관리 가능

`스코어보드 권한`

- `contest.scoreboard.view`: 스코어보드 조회 가능
- `contest.scoreboard.freeze`: 스코어보드 프리즈 설정 가능
- `contest.scoreboard.unfreeze`: 스코어보드 프리즈 해제 가능
- `contest.scoreboard.setting`: 스코어보드 공개 정책 설정 가능
- `contest.team_status.view`: 팀별 문제 풀이 상세 현황 조회 가능
- `contest.team_status.realtime_view`: 팀별 실시간 풀이 현황 조회 가능

`운영 권한`

- `contest.submission.view`: 해당 대회 제출 조회 가능
- `contest.submission.rejudge`: 해당 대회 재채점 요청 가능
- `contest.manager.invite`: 대회 매니저 초대 가능
- `contest.manager.permission_update`: 대회 매니저 권한 변경 가능
- `contest.manager.remove`: 대회 매니저 제거 가능

추가 원칙:

- 대회 운영자는 해당 대회의 모든 권한을 가진다.
- 대회 매니저는 부여받은 권한만 사용할 수 있어야 한다.
- 권한 없는 기능은 UI와 API에서 모두 차단해야 한다.
- 권한자 설정 페이지에서 권한 체크리스트 방식으로 관리할 수 있어야 한다.

### 서비스 매니저 권한 카탈로그 초안

서비스 마스터는 서비스 매니저를 추가/수정할 때 아래 권한을 개별적으로 부여할 수 있어야 한다.
필요하면 권한은 더 세분화 가능해야 하며, 최소한 아래 수준까지는 나뉘어야 한다.
각 권한은 `전역 적용` 또는 `특정 대회들에만 적용` 형태로 범위를 가질 수 있어야 한다.

`대회 생성/개최 권한`

- `contest.create`: 대회 생성 가능
- `contest.open`: 대회 오픈 가능
- `contest.close`: 대회 종료 처리 가능
- `contest.delete`: 대회 삭제 가능

`대회 기본 정보 권한`

- `contest.view`: 대회 목록 및 상세 조회 가능
- `contest.update_basic`: 대회 이름, 공개 여부, 상태 등 기본 정보 수정 가능
- `contest.update_schedule`: 시작 시각, 종료 시각, 예약 공개 시각 수정 가능
- `contest.update_organization`: 개최 기관, 주최 단체명 수정 가능

## 9. 기능 요구사항

### 9.1 대회 관리

- 서비스 마스터만 대회를 생성할 수 있어야 한다.
- 서비스 마스터만 대회를 오픈할 수 있어야 한다.
- 대회는 예약 공개 시각을 가질 수 있어야 한다.
- 대회는 시작 시각과 종료 시각을 가질 수 있어야 한다.
- 대회 공개 전에는 일반 참가 팀에게 노출되지 않거나 제한적으로 노출되어야 한다.
- 대회 목록에서는 현재 열려 있는 대회와 종료된 대회를 구분해 보여줄 수 있어야 한다.
- 대회 목록 또는 대회 정보에서는 어떤 단체가 대회를 열었는지 표시할 수 있어야 한다.
- 대회 상태는 최소한 아래 개념을 가져야 한다.
  - 준비중
  - 공개예정
  - 진행중
  - 종료됨

### 9.1.1 공지 및 긴급 안내

- 대회별 공지/안내 시스템이 필요하다.
- 대회에는 공지사항 게시판이 있어야 한다.
- 공지사항 게시판에서는 일반 공지, 규정 변경 공지, 운영 공지 등을 작성할 수 있어야 한다.
- 긴급 공지는 페이지 상단 네비게이션 바 영역에 강조 형태로 노출할 수 있어야 한다.
- 긴급 공지는 사용자가 직접 닫을 수 있어야 한다.
- 긴급 공지는 대회별로 노출 범위를 관리할 수 있어야 한다.
- 공지 작성/수정/삭제 권한은 대회 운영 권한에 따라 제어되어야 한다.

### 9.1.2 질문 게시판 / Clarification

- 대회별 질문 게시판이 필요하다.
- 참가 팀은 문제에 대해 질문을 등록할 수 있어야 한다.
- 운영진은 질문에 답변할 수 있어야 한다.
- 질문 게시판은 백준 스타일의 대회용 질문/답변 흐름을 참고한다.
- 질문은 공개/비공개 여부 또는 운영진만 볼 수 있는 형태를 고려할 수 있어야 한다.
- 질문 작성/답변/수정/숨김 처리 등의 권한은 역할에 따라 제어되어야 한다.

### 9.2 대회 운영자 관리

- 서비스 마스터는 대회별 운영자를 초대/추가/삭제할 수 있어야 한다.
- 대회 운영 권한은 대회 단위로 분리되어야 한다.
- 운영자는 자신이 담당하는 대회 범위 안에서 모든 관리 기능을 사용할 수 있어야 한다.
- 한 사용자는 여러 대회의 운영자를 겸할 수 있어야 한다.
- 서비스 마스터는 언제든지 모든 대회의 운영자를 직접 관리할 수 있어야 한다.
- 대회 운영자 추가 시 비밀번호 설정 메일을 발송할 수 있어야 한다.

### 9.3 대회 매니저 관리

- 대회 운영자는 해당 대회의 대회 매니저를 초대할 수 있어야 한다.
- 관련 권한이 있는 사용자는 해당 대회의 대회 매니저를 임명할 수 있어야 한다.
- 대회 운영자는 설정 페이지에서 대회 매니저의 권한을 관리할 수 있어야 한다.
- 대회 운영자는 설정 페이지에서 대회 매니저를 추방할 수 있어야 한다.
- 대회 매니저 권한은 대회 단위로 분리되어야 한다.
- 서비스 마스터는 모든 대회의 대회 매니저를 직접 관리할 수 있어야 한다.
- 대회 매니저 추가 시 비밀번호 설정 메일을 발송할 수 있어야 한다.
- 대회 매니저 권한은 개별 권한 단위로 세분화해 부여할 수 있어야 한다.
- 대회 매니저 권한 설정에는 최소한 아래 항목이 포함되어야 한다.
  - 대회 개요 변경 가능 여부
  - 대회 참가 팀 추가 가능 여부
  - 대회 문제 확인 가능 여부
  - 대회 스코어보드 확인 가능 여부
  - 대회 스코어보드 프리즈 가능 여부

### 9.4 서비스 매니저 관리

- 서비스 마스터는 서비스 매니저를 추가/수정/삭제할 수 있어야 한다.
- 서비스 마스터는 서비스 매니저의 권한을 변경할 수 있어야 한다.
- 서비스 매니저 관리는 전용 관리 페이지에서 수행할 수 있어야 한다.
- 서비스 매니저 권한은 전역 또는 특정 대회 범위 권한으로 설정할 수 있어야 한다.
- 서비스 매니저 추가 시 비밀번호 설정 메일을 발송할 수 있어야 한다.
- 서비스 매니저 권한은 개별 권한 단위로 세분화해 부여할 수 있어야 한다.
- 서비스 매니저 추가/수정 시 권한 체크리스트 또는 권한 그룹 형태로 권한을 설정할 수 있어야 한다.
- 서비스 매니저 추가/수정 시 각 권한의 적용 범위를 `전역` 또는 `특정 대회들`로 지정할 수 있어야 한다.
- 서비스 매니저 권한은 추후 증가해도 관리 가능하도록 확장 가능한 구조여야 한다.

### 9.5 참가 팀 관리

- 대회 운영자는 해당 대회 참가 팀을 추가/수정/삭제할 수 있어야 한다.
- 서비스 마스터는 모든 대회의 참가 팀을 직접 추가/수정/삭제할 수 있어야 한다.
- 참가 팀 정보는 대회 단위로 관리 가능해야 한다.
- 참가 팀 등록 방식은 수동 관리가 가능해야 한다.
- 추후 일괄 등록 기능을 고려할 수 있도록 구조를 잡는다.
- 참가 팀 등록 시 팀명을 저장할 수 있어야 한다.
- 참가 팀 등록 시 대표 연락용 이메일을 저장할 수 있어야 한다.
- 참가 팀 등록 시 팀 로그인 식별자 목록을 저장할 수 있어야 한다.
- 참가 팀 등록 시 참가 상태를 저장할 수 있어야 한다.
- 참가 팀 등록 시 팀 접속 정책과 동시 접속 가능 수를 저장할 수 있어야 한다.
- 팀 로그인 식별자와 팀명을 연결된 데이터로 관리해야 한다.
- 개인 참가도 1인 1팀으로 등록해야 한다.
- 동일 대회의 운영자 또는 매니저는 참가 팀으로 등록할 수 없어야 한다.

참가 팀 데이터 최소 범위:

- 팀명
- 대표 연락용 이메일
- 로그인 식별자 목록
- 참가 상태
- 팀 접속 정책
- 동시 접속 가능 수

팀 상태 모델은 운영 상태와 로그인 이력을 분리한다.

참가 팀 운영 상태 enum:

- `invited`: 팀이 등록되었고 아직 로그인 가능한 준비 상태
- `active`: 팀이 정상적으로 참가 가능한 상태
- `disqualified`: 실격 처리된 상태
- `disabled`: 운영상 비활성화된 상태

로그인/인증 이력 필드 예시:

- `last_login_at`
- `last_otp_verified_at`
- `current_session_id`

참고:

- 팀의 `운영 상태`와 `로그인 이력`은 별도 필드로 관리한다.
- 즉, `logged_in` 같은 값을 상태 enum에 넣지 않고, 로그인 이력 필드로 관리한다.
- 이렇게 해야 실격/비활성/초대 상태와 로그인 기록이 섞이지 않아 운영에 유리하다.

참가 팀 데이터 비포함 범위:

- 팀원 프로필 상세 정보
- 학교/학년 등 확장 프로필
- 기타 내부 운영용 팀 정보

단, 멤버별 동시 접속이 필요한 대회에서는 최소한 로그인 식별자 수준의 팀원 정보는 이 서비스에서 관리해야 한다.

참가 팀 관리 상세는 추후 더 구체화한다. 현재 기준으로는 팀 사전 등록 방식이 필요하다.

### 9.5.1 팀 인증 및 세션 정책

- 대표 연락용 이메일 변경은 권한 있는 사람만 수행할 수 있어야 한다.
- 팀 로그인 정책은 대회별로 설정 가능해야 한다.
- 최소한 `대표자 단일 접속 모드`와 `팀원 다중 접속 모드`를 지원할 수 있어야 한다.
- 같은 브라우저 세션의 여러 탭은 허용해야 한다.
- 팀원 다중 접속 모드에서는 설정된 동시 접속 가능 수만큼 활성 세션을 허용해야 한다.
- 예를 들어 `3인 1팀` 대회에서는 팀원 3명이 동시에 접속해도 정상 동작해야 한다.
- 동시 접속 가능 수를 초과하는 로그인 시도에 대해서는 차단 또는 세션 전환 정책을 둘 수 있어야 한다.
- 대표자 단일 접속 모드인 대회에서는 세션 전환 정책을 적용할 수 있어야 한다.
- 같은 기기 내 같은 브라우저 세션에서는 로그인 상태가 유지되어야 한다.

### 9.6 문제 관리

- 대회 운영자는 문제를 추가/수정/삭제할 수 있어야 한다.
- 문제 순서를 정렬할 수 있어야 한다.
- 문제 이름을 관리할 수 있어야 한다.
- 문제 번호 또는 문제 표시 순서를 관리할 수 있어야 한다.
- 문제 본문을 관리할 수 있어야 한다.
- 문제 입력 설명을 관리할 수 있어야 한다.
- 문제 출력 설명을 관리할 수 있어야 한다.
- 문제 제한값을 관리할 수 있어야 한다.
  - 시간 제한
  - 메모리 제한
  - 입력 범위 설명
  - 기타 제약 조건
- 예제 입력/출력을 여러 개 등록할 수 있어야 한다.
- 문제에 필요한 리소스를 업로드/관리할 수 있어야 한다.
- 채점용 테스트케이스를 수동으로 추가할 수 있어야 한다.
- 테스트케이스 제너레이터를 등록하고 관리할 수 있어야 한다.

### 9.6.1 문제 공개 범위

- 대회 운영자와 권한 있는 대회 매니저는 대회 시작 전/진행 중/종료 후 언제든 문제를 확인할 수 있어야 한다.
- 참가 팀은 대회가 시작된 이후에만 문제를 볼 수 있어야 한다.
- 미로그인 사용자는 대회 시작 전과 대회 진행 중에는 문제를 볼 수 없어야 한다.
- 대회 종료 후 미로그인 사용자에게 문제를 공개할지 여부는 대회 설정 옵션으로 둘 수 있어야 한다.

### 9.6.2 제출 코드 공개 범위

- 참가 팀은 자신의 제출 코드를 언제든 확인할 수 있어야 한다.
- 대회 운영자와 권한 있는 대회 매니저는 모든 제출 코드를 언제든 확인할 수 있어야 한다.

### 9.6.3 대회 중 문제 수정 정책

- 대회 진행 중에는 문제 본문 수정이 불가능해야 한다.
- 대회 진행 중에는 문제 데이터와 규칙의 변경을 금지하는 것을 기본 정책으로 한다.

### 9.6.4 문제 리소스 정책

- 문제 리소스는 문제 본문에 들어가는 이미지 파일만 허용한다.
- 허용 확장자는 최소한 아래를 포함해야 한다.
  - png
  - jpg
  - jpeg
  - webp
- 이미지 파일 크기 제한은 파일당 1GB로 한다.
- 허용 확장자와 업로드 크기 제한은 시스템 설정으로 조정 가능하면 좋다.

### 9.7 제출 및 결과 확인

- 참가 팀은 문제를 제출할 수 있어야 한다.
- 한 문제는 아래 4개 언어로 모두 제출 가능해야 한다.
  - C99
  - C++17
  - Python 3.13
  - Java 8
- 제출은 대회와 문제와 팀에 연결되어 저장되어야 한다.
- 참가 팀은 자신의 제출 목록을 조회할 수 있어야 한다.
- 참가 팀은 자신의 제출 상세와 판정 결과를 조회할 수 있어야 한다.
- 팀 기준으로 문제별 풀이 현황을 볼 수 있어야 한다.
- 채점 중인 제출은 상태 갱신이 가능해야 한다.

### 9.7.1 운영자/매니저용 팀별 풀이 현황 조회

- 대회 운영자와 관련 권한이 있는 대회 매니저는 팀별 문제 풀이 상태를 조회할 수 있어야 한다.
- 해당 화면은 실시간으로 갱신되어야 한다.
- 팀별로 최소한 아래 정보를 확인할 수 있어야 한다.
  - 어떤 문제를 풀었는지
  - 각 문제의 해결 여부
  - 문제별 정답 시각
  - 문제별 마지막 제출 시각
  - 문제별 제출 횟수
  - 문제별 오답 횟수
  - 문제별 페널티
  - 현재 등수
  - 총 해결 문제 수
  - 총 페널티

### 9.7.2 스코어보드 및 랭킹 규칙

- 스코어보드와 랭킹 규칙은 기본적으로 백준 온라인 저지(BOJ) 대회 방식을 따른다.
- 현재 기준 대회 유형은 `일반 대회`만 지원한다.
- 즉, 최종 점수는 `맞은 문제 수` 기준으로 계산한다.
- 부분 점수 문제는 지원하지 않는다.
- 문제 결과는 기본적으로 일반 문제 기준으로 판정한다.
- 스코어보드 정렬은 아래 우선순위를 따른다.
  - 더 많은 문제를 맞은 팀이 우선
  - 맞은 문제 수가 같으면 총 페널티가 적은 팀이 우선
  - 맞은 문제 수와 총 페널티가 같으면 마지막 문제 해결 시각이 빠른 팀이 우선
- 기본 페널티 계산은 아래 공식을 따른다.
  - `맞은 시각 + (정답 제출 전 오답 횟수 * 20분)`
- 여기서 `맞은 시각`과 문제별 제출 시각 계산의 기준은 `실제 제출 생성 시각(submitted_at)`이어야 한다.
- 즉, 채점 완료 시각이나 큐에서 실제 처리된 시각은 랭킹과 페널티 계산 기준으로 사용하지 않는다.
- 컴파일 에러는 페널티 계산에서 제외하는 것을 대회 기본 규정으로 한다.
- 오답 페널티는 해당 문제가 최종적으로 맞았을 때만 누적 반영한다.
- 대회별로 스코어보드 프리즈를 설정할 수 있어야 한다.
- 프리즈 시작 시각은 대회 설정에서 조정할 수 있어야 한다.
- 프리즈 중에는 공개 스코어보드 갱신을 제한할 수 있어야 한다.
- 프리즈 중 제출은 내부적으로 계속 채점되더라도 공개 스코어보드에는 노란색 또는 미공개 상태로 표시할 수 있어야 한다.
- 프리즈 이후 정답 제출은 최종 공개 전까지 공개 스코어보드에 반영되지 않아야 한다.
- 프리즈 해제 및 최종 스코어보드 공개는 대회 종료 즉시 자동 공개하지 않고, 관리자가 명시적으로 공개해야 한다.
- 운영자와 권한 있는 매니저는 프리즈 여부와 관계없이 내부 현황을 더 상세히 볼 수 있어야 한다.
- 재채점은 제출을 다시 채점하는 운영 기능을 의미한다.
- 대회가 진행 중인 동안에는 데이터 추가 재채점을 할 수 없어야 한다.
- 대회 종료 후 대회 제출 재채점은 가능해야 한다.
- 대회 종료 후 재채점을 수행하더라도 대회 당시 스코어보드는 자동으로 변경되지 않아야 한다.
- 대회 스코어보드를 변경해야 하는 경우에는 별도 운영 절차가 필요해야 한다.

### 9.7.3 채점 결과 enum 및 출력 비교 규칙

- 채점 결과 enum은 아래 상태를 기준으로 확정한다.
  - `waiting`: 기다리는 중
  - `preparing`: 채점 준비 중
  - `judging`: 채점 중
  - `accepted`: 맞았습니다
  - `presentation_error`: 출력 형식이 잘못되었습니다
  - `wrong_answer`: 틀렸습니다
  - `time_limit_exceeded`: 시간 초과
  - `memory_limit_exceeded`: 메모리 초과
  - `output_limit_exceeded`: 출력 초과
  - `runtime_error`: 런타임 에러
  - `compile_error`: 컴파일 에러
  - `system_error`: 시스템 에러
- 출력 비교 규칙은 `출력 형식이 잘못되었습니다` 규칙을 지원해야 한다.
- 줄 앞의 의미 없는 공백, 의미 없는 빈 줄, 출력 형식 불일치 등은 `presentation_error`로 판정할 수 있어야 한다.
- 줄 뒤의 공백은 무시하는 방향을 기본 규칙으로 한다.
- 출력 값 자체가 정답과 다르면 `wrong_answer`로 판정한다.

### 9.8 인증 및 로그인

인증 및 로그인은 크게 두 가지 흐름으로 나뉜다.

#### 9.8.1 서비스 마스터 / 서비스 매니저 / 대회 운영자 / 대회 매니저 로그인

- 서비스 마스터, 서비스 매니저, 대회 운영자, 대회 매니저는 이메일 + 비밀번호 기반 로그인 기능이 필요하다.
- 서비스 매니저 및 대회 운영자 및 대회 매니저는 권한 있는 사용자가 추가했을 때 계정 활성화 메일 또는 비밀번호 설정 메일을 받아야 한다.
- 해당 메일을 통해 최초 비밀번호를 설정한 뒤 로그인할 수 있어야 한다.
- 비밀번호 재설정 기능이 있어야 한다.
- 비밀번호를 잊은 경우 이메일로 재설정 메일을 발송해 새 비밀번호를 설정할 수 있어야 한다.
- 최초 비밀번호 설정 링크와 비밀번호 재설정 링크는 만료 시간을 가져야 한다.
- 권한자용 로그인은 일반 참가 팀 로그인과 분리된 흐름이어야 한다.

#### 9.8.2 참가 팀 로그인

- 참가 팀 로그인은 이메일 인증 기반으로 동작해야 한다.
- 로그인 이메일은 대회 참가 팀으로 사전에 등록된 팀 로그인 식별자여야 한다.
- 팀원 다중 접속 모드에서는 팀원별로 등록된 이메일 또는 식별자를 사용해 로그인할 수 있어야 한다.
- 시스템은 해당 이메일로 인증번호를 발송해야 한다.
- 팀은 인증번호를 입력해 로그인해야 한다.
- 로그인은 팀의 대회 참가 자격과 연결되어야 한다.
- 팀명과 로그인 식별자 등 사전 등록 정보는 로그인 후 식별 가능해야 한다.
- 로그인 후 현재 대회의 팀 접속 정책과 활성 세션 수를 인지할 수 있어야 한다.

#### 9.8.3 공통 인증 요구사항

- 정확한 세션 방식과 만료 정책은 추후 확정한다.
- 메일 발송 실패, 만료 토큰, 잘못된 인증번호 등에 대한 예외 처리가 필요하다.
- 권한자 로그인과 참가 팀 로그인은 화면과 정책이 명확히 분리되어야 한다.

### 9.8.4 메일 정책

- 초대 메일, OTP 메일, 비밀번호 재설정 메일 발송 정책이 필요하다.
- 각 메일은 만료 시간을 가져야 한다.
- OTP는 재발송 제한과 시도 횟수 제한이 필요하다.
- 비밀번호 재설정 메일은 재발송 제한이 필요하다.
- 메일 발송 실패 시 사용자에게 재시도 가능한 오류 상태를 제공해야 한다.
- 메일 발송 관련 rate limit 과 abuse 방지 정책이 필요하다.

### 9.8.5 OTP 정책

- OTP는 숫자 6자리로 구성한다.
- OTP 유효 시간은 10분으로 한다.
- OTP 관련 rate limit은 10분당 100회 기준으로 제한한다.
- OTP 검증 실패 횟수 제한도 필요하지만, 대회 중 정상 사용자를 과도하게 막지 않도록 완화된 정책으로 설정해야 한다.

- 로그인 페이지

- 참가 팀용 로그인 페이지

- 비밀번호 재설정 요청 페이지
- 비밀번호 재설정 완료 페이지
- 대회 목록 페이지
- 대회 메인 또는 대회 홈 페이지
- 문제 목록 페이지
- 문제 상세 페이지
- 제출 페이지
- 제출 현황 페이지
- 제출 상세 페이지
- 도움말 페이지
- 운영자용 대회 관리 페이지
- 운영자용 문제 관리 페이지
- 운영자용 참가 팀 관리 페이지
- 운영자용 테스트케이스/리소스 관리 페이지
- 운영자용 대시보드 페이지
- 공개 채점 서버 현황 페이지
- 대회 공지사항 게시판 페이지
- 질문 게시판 페이지
- 서비스 마스터용 대회 생성/설정 페이지
- 서비스 마스터용 서비스 매니저 관리 페이지
- 서비스 마스터용 채점 서버 설정 페이지

프론트 동작 요구사항:

- 메인 페이지가 필요하다.
- 로그인 여부에 따라 접근 가능한 화면을 구분해야 한다.
- 역할에 따라 접근 가능한 메뉴와 기능이 달라야 한다.
- 참가 팀의 대회 참가 흐름이 단순해야 한다.
- 운영자는 문제/참가 팀/리소스를 관리할 수 있어야 한다.
- 채점 상태는 사용자가 갱신 결과를 확인할 수 있어야 한다.
- 대회 시작 전/진행 중/종료 후 상태에 따라 UI가 달라져야 한다.

메인 페이지 요구사항:

- `Zerone Online Judge` 소개가 있어야 한다.
- 운영 방식 소개가 있어야 한다.
- 대회 개최 방식 소개가 있어야 한다.
- 대회 규정 안내가 있어야 한다.
- 대회 생성 요청 방법을 안내해야 한다.
- 메일로 대회 생성 요청하는 방법을 설명해야 한다.
- 요청 메일 템플릿을 제공해야 한다.

대회 운영 규정집 페이지 요구사항:

- 대회 운영 규정을 정리한 전용 페이지가 필요하다.
- 랭킹 규칙, 페널티 규칙, 프리즈 규칙, 재채점 규칙 등을 확인할 수 있어야 한다.
- 팀 구성 규칙, 팀 로그인 정책, 동시 접속 정책도 확인할 수 있어야 한다.
- 참가 팀과 운영진 모두 접근 가능해야 한다.

대회 페이지 요구사항:

- 현재 열려 있는 대회 목록을 볼 수 있어야 한다.
- 과거에 열렸던 대회 목록도 볼 수 있어야 한다.
- 각 대회에 대해 주최 단체 이름을 표시할 수 있어야 한다.
- 특정 대회를 선택하면 대회 개요를 볼 수 있어야 한다.
- 참가하려는 사용자는 해당 대회의 로그인 흐름으로 들어갈 수 있어야 한다.
- 대회 공지사항 게시판으로 이동할 수 있어야 한다.
- 질문 게시판으로 이동할 수 있어야 한다.

참가 팀 로그인 화면 요구사항:

- 대회 참가 팀으로 등록된 로그인 이메일 또는 식별자 입력
- 인증번호 발송
- 인증번호 확인
- 필요 시 현재 활성 세션/동시 접속 가능 수 안내
- 로그인 완료

권한자 로그인 화면 요구사항:

- 이메일 입력
- 비밀번호 입력
- 최초 비밀번호 설정 진입
- 비밀번호 재설정 요청
- 재설정 메일을 통한 비밀번호 변경

서비스 매니저 관리 화면 요구사항:

- 서비스 매니저 목록 조회
- 서비스 매니저 추가
- 서비스 매니저 수정
- 서비스 매니저 삭제
- 서비스 매니저별 권한 체크리스트 제공
- 서비스 매니저별 권한 범위 설정 기능 제공
- 특정 권한을 어느 대회들에 적용할지 선택하는 UI가 필요하다
- 서비스 매니저별 권한 변경 이력 확인 여부는 추후 결정

대회 매니저 권한 설정 화면 요구사항:

- 대회 매니저 목록 조회
- 대회 매니저 초대
- 대회 매니저 삭제 및 추방
- 대회 매니저별 권한 체크리스트 제공
- 대회 매니저별로 대회 개요 변경, 참가 팀 추가, 문제 조회, 스코어보드 조회, 스코어보드 프리즈 등의 권한을 설정할 수 있어야 한다

운영자 대시보드 화면 요구사항:

- 대회 진행 중 제출량을 한눈에 볼 수 있어야 한다.
- 현재 채점 대기열 상태를 볼 수 있어야 한다.
- 문제별 정답 수와 제출 수를 볼 수 있어야 한다.
- 팀별 현재 상태와 등수 변화를 볼 수 있어야 한다.
- 실시간 또는 준실시간으로 갱신되어야 한다.

공지/질문 게시판 화면 요구사항:

- 공지사항 목록/상세/작성/수정/삭제 기능이 필요하다.
- 긴급 공지는 상단 네비게이션 바에 강조 표시되어야 한다.
- 질문 게시판은 질문 등록, 목록 조회, 상세 조회, 답변 작성이 가능해야 한다.

공개 채점 서버 현황 페이지 요구사항:

- 비로그인 사용자도 접근 가능해야 한다.
- 현재 운영 중인 채점 서버 수를 볼 수 있어야 한다.
- 서버별 현재 처리 중 작업 수 또는 큐 수를 볼 수 있어야 한다.
- 채점 작업 할당 규칙이나 현재 분산 상태를 확인할 수 있어야 한다.
- 모니터링 목적의 기본 상태 정보를 제공해야 한다.

프론트에서 추가 논의가 필요한 항목:

- 정확한 디자인 시스템 범위
- 서비스 마스터/운영자 화면의 정보 구조
- 제출 상태 갱신 방식을 폴링으로 시작할지 SSE로 갈지
- 문제 본문 포맷 방식

## 11. 백엔드 요구사항

백엔드는 FastAPI 기반이며, 단순 제출 API 서버가 아니라 플랫폼 중심 서버가 되어야 한다.
또한 공개 웹/API 계층과 비공개 채점 계층을 명확히 분리하는 구조여야 한다.
그리고 내부 설계 방식은 `Domain Driven Design(DDD)`를 기본 방향으로 삼는다.

### 11.0 DDD 문서

- DDD 상세 초안은 [../07-architecture/ddd.md](../07-architecture/ddd.md)에 분리한다.
- 메인 요구사항 문서에는 제품 요구사항과 운영 정책 중심 내용만 남긴다.
- bounded context, aggregate, 이벤트, layered architecture 상세는 별도 DDD 문서를 기준으로 갱신한다.

### 11.1 백엔드 배치 구조

백엔드는 아래 두 계층으로 나뉘는 것을 기본 전제로 한다.

- `웹/API 서버`
  - 공개 인터넷에서 접근 가능한 서버
  - 프론트엔드, 운영 화면, 팀 로그인, 권한자 로그인, 대회/문제/팀/권한 관리 API를 제공
  - 제출을 접수하고 저장한 뒤 채점 요청을 생성
  - 채점 결과를 수신하거나 반영 상태를 동기화
- `채점 서버 계층`
  - 공개 인터넷에 직접 노출되지 않는 비공개 서버
  - Tailscale로 연결된 사설망 내부에서만 통신
  - 실제 컴파일/실행/채점을 담당
  - Docker 샌드박스 내부에서 언어별 실행 환경을 제공

배치 원칙:

- 사용자는 웹/API 서버에만 직접 접근해야 한다.
- 채점 서버는 Tailscale 네트워크에 가입된 노드로만 운영한다.
- 웹/API 서버는 채점 서버와 직접 또는 채점 큐/디스패처를 통해 통신하되, 통신 경로는 Tailscale 사설망으로 제한해야 한다.
- 채점 서버 관리, 상태 수집, 작업 분산도 동일하게 Tailscale 사설망을 기준으로 동작해야 한다.

### 11.1.1 권장 서비스 분리 구조

현재 기준으로는 단순히 `웹`과 `채점 서버`만 둘 것이 아니라, 아래 3개 실행 단위를 분리하는 방향이 가장 안정적이다.

- `web/api`
  - 공개 인터넷에 노출되는 FastAPI 서버
  - 사용자 요청 처리, 인증, 권한, 대회/문제/팀/제출 API 담당
  - 제출 저장 후 채점 작업 생성
- `judge-dispatcher`
  - 비공개 내부 서비스
  - 전역 FIFO 큐 관리
  - 채점 서버 heartbeat 수집
  - 가용 slot 기준 작업 할당
  - lease 만료, 재시도, 장애 전환 처리
- `judge-agent`
  - 각 채점 서버 VM에서 실행되는 에이전트
  - dispatcher에 heartbeat 전송
  - 가용 slot이 있으면 작업 요청
  - Docker 샌드박스 안에서 실제 컴파일/실행/채점 수행

운영 구역 관점에서는 아래처럼 본다.

- `공개 구역`
  - web/api
- `비공개 Tailscale 구역`
  - judge-dispatcher
  - judge-agent

즉, 논리적으로는 3개 서비스지만, 외부에 공개되는 건 `web/api` 하나뿐이다.

### 11.1.3 왜 dispatcher를 분리하는가

- web/api가 직접 채점 서버와 세션을 오래 유지하지 않도록 하기 위함
- 사용자 요청 처리와 채점 스케줄링을 분리해 API 응답 지연을 줄이기 위함
- heartbeat, lease, 재시도, 장애 전환 로직을 독립 서비스로 격리하기 위함
- 추후 judge 수가 늘어나도 web/api를 크게 건드리지 않기 위함
- 공개망 서버와 비공개 채점망 사이 제어 지점을 하나로 모으기 위함

백엔드 핵심 책임:

- 인증과 권한 처리
- 사용자 및 역할 관리
- 서비스 매니저 관리
- 대회 관리
- 대회 운영자 관리
- 대회 매니저 관리
- 대회 참가 팀 관리
- 문제 관리
- 문제 리소스 관리
- 공지사항 관리
- 질문 게시판 관리
- 감사 로그 조회
- 운영자 대시보드 제공
- 공개 채점 서버 상태 제공
- 예제 입출력 관리
- 테스트케이스 및 제너레이터 관리
- 제출 생성 및 조회
- 채점 요청 생성
- 채점 결과 반영
- 채점 서버/채점 클러스터 설정 관리
- Tailscale 기반 채점 서버 라우팅 및 상태 관리

예상 API 범주:

- Auth
- Users / Me
- Roles / Permissions
- Service Managers
- Contests
- Contest Managers
- Contest Staff
- Contest Participants
- Problems
- Problem Resources
- Problem Examples
- Testcases
- Testcase Generators
- Submissions
- Submission Results
- Judge Queue
- Judge Servers
- Judge Cluster Settings
- Audit Logs

백엔드가 보장해야 하는 방향:

- 권한 체크는 엄격해야 한다.
- 권한 시스템은 세분화된 permission 기반 RBAC 구조여야 한다.
- permission에는 적용 범위(scope) 개념이 포함되어야 한다.
- 대회 단위 데이터 경계가 명확해야 한다.
- 동일 대회 내 역할 충돌 규칙을 강제해야 한다.
- 서비스 마스터는 모든 권한 검사를 우회하는 최상위 권한이어야 한다.
- 대회 운영자는 자신이 담당하는 대회의 모든 permission을 가진 것으로 처리해야 한다.
- 제출 생성과 채점 요청 생성은 안정적으로 연결되어야 한다.
- 제출 저장과 채점 요청 생성 사이에는 유실 방지 장치가 필요하다.
- 채점 시스템 장애가 있어도 제출 데이터 자체는 유실되지 않아야 한다.
- 채점 서버는 외부 공개망이 아니라 Tailscale 사설망을 통해서만 접근해야 한다.
- 웹/API 서버는 채점 서버의 Tailscale 식별자, 연결 상태, 할당 가능 여부를 관리할 수 있어야 한다.
- 운영자가 채점 서버 상태와 설정을 볼 수 있어야 한다.
- 권한자용 비밀번호 설정 메일과 비밀번호 재설정 메일 발송을 지원해야 한다.
- 이메일 인증번호 발송과 검증 흐름을 지원해야 한다.
- 대회 정책에 따라 팀 대표 또는 팀원별 로그인 흐름을 지원해야 한다.
- 누가 언제 어떤 설정, 문제, 참가 팀, 권한, 스코어보드를 변경했는지 감사 로그로 남길 수 있어야 한다.

### 11.2 감사 로그 요구사항

- 감사 로그 기능은 필수다.
- 최소한 아래 작업은 로그로 기록되어야 한다.
  - 대회 생성/수정/삭제/오픈/종료
  - 공지사항 작성/수정/삭제
  - 질문 답변 작성/수정
  - 운영자/매니저/서비스 매니저 권한 변경
  - 참가 팀 추가/수정/삭제/실격 처리
  - 문제/테스트케이스/리소스 변경
  - 스코어보드 프리즈 설정/해제/최종 공개
  - 채점 서버 설정 변경
- 감사 로그에는 최소한 작업자, 시각, 대상, 작업 종류, 변경 요약이 포함되어야 한다.

### 11.3 운영/인프라 요구사항

- 백업 및 복구 요구사항이 필요하다.
- 모니터링 및 알람 체계가 필요하다.
- 큐 재시도, 중복 처리, idempotency 정책이 필요하다.
- 채점 서버 장애 시 작업 재할당이 가능해야 한다.
- Tailscale 연결 상태 이상 시 해당 채점 서버를 자동으로 비할당 처리할 수 있어야 한다.
- 메일 서버 장애 대응 정책이 필요하다.
- 권한 변경, 문제 수정, 실격 처리 같은 민감 작업에 대한 로그를 장기간 보관할 수 있어야 한다.
- 채점 서버 현황, 서버별 처리량, 할당 상태를 모니터링할 수 있어야 한다.
- 운영자는 채점 서버별 Tailscale 연결 상태를 확인할 수 있어야 한다.

## 12. 채점 시스템 요구사항

채점기는 백엔드와 별도로 동작하는 독립 시스템이어야 한다.

핵심 요구사항:

- 백엔드가 제출 요청을 받으면 채점 작업 큐에 넣을 수 있어야 한다.
- 큐는 FIFO 방식으로 동작해야 한다.
- FIFO 기준은 `제출 생성 시각(submitted_at)` 순서여야 한다.
- 트래픽이 많이 몰려도 안정적으로 버텨야 한다.
- 로드 밸런싱을 통해 여러 채점 서버로 작업을 분산할 수 있어야 한다.
- 채점 서버 수를 조정할 수 있어야 한다.
- 채점 서버를 동적으로 늘리거나 줄이는 운영 기능을 고려해야 한다.
- 백엔드 또는 서비스 마스터 기능을 통해 채점 서버 셋팅을 관리할 수 있어야 한다.
- 채점 서버는 Tailscale로 연결된 서버들로 구성한다.
- 웹/API 서버는 Tailscale 네트워크를 통해서만 채점 서버에 작업을 전달해야 한다.
- 채점 서버는 외부 공인 주소를 필수로 가지지 않아도 운영 가능해야 한다.
- 채점 디스패치 계층은 서버별 가용 상태, 활성 작업 수, 큐 길이, 연결 상태를 기준으로 분산할 수 있어야 한다.

### 12.1 권장 큐 처리 흐름

현재 요구사항 기준으로 가장 적절한 흐름은 아래와 같다.

1. `web/api`가 제출 요청을 받는다.
2. 제출 본문, 언어, 팀, 문제, `submitted_at`을 저장한다.
3. 같은 트랜잭션 또는 안정적으로 연결된 후속 처리로 `judge_job`을 생성한다.
4. `judge_job`의 초기 상태는 `waiting`이다.
5. `judge-dispatcher`는 `submitted_at ASC, submission_id ASC` 기준으로 전역 FIFO 순서를 유지한다.
6. 각 `judge-agent`는 주기적으로 heartbeat를 보내고 현재 남은 slot 수를 보고한다.
7. dispatcher는 slot이 있는 judge-agent에게 가장 오래 기다린 작업을 lease로 할당한다.
8. judge-agent는 제출 코드, 문제 설정, 테스트케이스를 받아 채점을 시작한다.
9. 채점 중 상태는 `preparing`, `judging` 등으로 갱신된다.
10. 최종 결과는 `accepted`, `wrong_answer`, `compile_error` 등으로 저장된다.
11. web/api는 최종 결과를 기반으로 제출 조회 화면, 스코어보드, 운영 대시보드를 갱신한다.

중요 원칙:

- 스코어보드와 페널티 계산 기준 시각은 `judged_at`이 아니라 `submitted_at`이다.
- 실제 채점이 늦게 끝나더라도 순위 계산 기준은 제출 시각을 사용한다.
- 큐 순서는 전역 FIFO이며, 기본적으로 대회/문제별 우선순위는 두지 않는다.

### 12.2 lease / heartbeat / 재할당 정책

- judge-agent는 일정 주기로 heartbeat를 보내야 한다.
- heartbeat에는 최소한 아래 정보가 포함되어야 한다.
  - `node_id`
  - `tailscale_ip` 또는 내부 식별자
  - `status`
  - `total_slots`
  - `free_slots`
  - `running_jobs`
  - `last_seen_at`
- dispatcher는 일정 시간 heartbeat가 끊긴 node를 `unavailable`로 표시해야 한다.
- unavailable node에 lease된 작업은 lease 만료 후 자동 재할당할 수 있어야 한다.
- 같은 작업이 중복 실행되지 않도록 `lease_token`과 `job_version` 또는 동등한 idempotency 장치가 필요하다.
- 이미 최종 결과가 반영된 작업은 중복 반영되지 않아야 한다.

### 12.3 judge-agent slot 모델

- judge-agent는 VM 한 대당 고정된 동시 실행 slot 수를 가진다.
- slot은 `컴파일 + 실행 + 비교` 전체 파이프라인을 포함하는 단위다.
- slot 수는 CPU뿐 아니라 메모리, 언어별 피크 사용량, Docker 오버헤드를 고려해 설정해야 한다.
- 각 judge VM의 실제 slot 수는 운영자가 조정 가능해야 한다.
- dispatcher는 free slot이 0인 node에는 새 작업을 할당하지 않아야 한다.

### 12.4 권장 Proxmox 배치 초안

현재 자원 기준에서 시작점으로는 아래 구성이 적절하다.

- `web/api` VM: `4 vCPU / 8~12GB RAM`
- `db` VM: `4 vCPU / 16~24GB RAM`
- `redis/queue` 또는 내부 메시징 VM: `2~4 vCPU / 8GB RAM`
- `judge-dispatcher` VM: `2~4 vCPU / 4~8GB RAM`
- `judge` VM `5대`: 각 `6 vCPU / 20~24GB RAM`

운영 시작 기준 권장:

- judge VM당 동시 slot `4`부터 시작
- 전체 동시 채점 `20` 정도로 초기 운영
- 부하 테스트 후 `slot 5` 이상 증설 여부 판단

추가 원칙:

- Proxmox 호스트 자체를 위해 CPU와 메모리를 일부 반드시 남겨둔다.
- judge VM을 너무 크게 1대에 몰지 않고 여러 대로 나눈다.
- 한 judge VM 장애가 전체 채점 중단으로 이어지지 않게 한다.
- judge VM은 공개망에 직접 노출하지 않는다.

채점 서버 실행 요구사항:

- 각 제출은 Docker 샌드박스 내부에서 실행해야 한다.
- 네트워크 패킷 송수신은 차단해야 한다.
- 호스트 서버 접근은 차단해야 한다.
- 컨테이너 권한은 최소화해야 한다.
- 시간 제한 초과 시 즉시 결과를 도출해야 한다.
- 메모리 제한 초과 시 즉시 결과를 도출해야 한다.
- 빠른 채점 성능과 보안을 동시에 만족해야 한다.
- 채점 서버는 Tailscale 통신 외의 불필요한 외부 인바운드 접근을 허용하지 않는 방향으로 구성해야 한다.

지원 언어:

- C99
- C++17
- Python 3.13
- Java 8

판정 기준:

- 문제에 정의된 테스트케이스를 기준으로 채점한다.
- 수동 추가 테스트케이스와 생성기 기반 테스트케이스를 모두 사용할 수 있어야 한다.
- 결과 상태는 백준 시스템을 따른다.

현재 기준 최소 결과 상태 후보:

- 채점 준비 중

- 맞았습니다

- 컴파일 에러
- 런타임 에러

- 서비스 매니저 역할이 존재한다.
- 서비스 매니저는 개별 permission 묶음으로 권한을 부여받는다.

- 대회 매니저 역할이 존재한다.
- 대회 운영자는 해당 대회 전체 권한을 가진다.
- 대회 매니저는 해당 대회의 세분화된 permission 묶음을 부여받는다.
- 서비스 마스터는 대회를 개최하고 예약 공개할 수 있다.
- 서비스 마스터만 대회를 개최할 수 있다.
- 서비스 마스터만 대회를 오픈할 수 있다.
- 서비스 마스터는 서비스 내 모든 것을 관리할 수 있다.
- 서비스 마스터는 서비스 매니저를 추가/수정/삭제하고 권한 변경할 수 있다.
- 서비스 매니저 권한은 대회 생성, 대회 수정, 대회 삭제, 대회 상세 수정, 운영자/매니저/참가 팀/문제 관리 등으로 세분화한다.
- 서비스 매니저 권한은 전역 또는 특정 대회들에 한정해 부여할 수 있다.
- 서비스 마스터는 대회별 운영자를 초대할 수 있다.

- 개인 참가도 1인 1팀으로 처리한다.
- 일부 대회는 `3인 1팀`을 포함한 다인 팀을 지원할 수 있다.
- 일부 대회는 팀원 다중 접속을 지원해야 하며, `3인 1팀` 대회에서는 3명 모두 동시에 접속할 수 있어야 한다.

- 팀명과 로그인 식별자는 사전에 연결되어 등록된다.
- 팀 상태는 운영 상태와 로그인 이력을 분리해 관리한다.
- 팀 접속 정책은 대회별로 관리한다.
- 같은 브라우저 세션의 여러 탭은 허용한다.
- 스코어보드 규칙은 기본적으로 BOJ 일반 대회 방식을 따른다.
- 부분 점수 문제는 지원하지 않는다.
- 문제는 대회 진행 중 수정할 수 없다.
- 운영진과 권한 있는 매니저는 언제든 문제와 제출 코드를 볼 수 있다.
- 참가 팀은 자신의 제출 코드를 언제든 볼 수 있다.
- 미로그인 사용자의 종료 후 문제 공개는 옵션으로 제공한다.
- 공지사항 게시판과 질문 게시판이 필요하다.
- 긴급 공지는 상단 네비게이션 바에서 강조 표시되어야 한다.
- 감사 로그 기능이 필수다.
- 운영자 대시보드가 필요하다.
- 대회 운영 규정집 페이지가 필요하다.
- 공개 채점 서버 현황 페이지가 필요하며, 비로그인 사용자도 볼 수 있어야 한다.
- 채점 결과는 BOJ 스타일 상태 체계를 따른다.
- 컴파일 에러는 페널티에서 제외한다.
- 프리즈 시작 시각은 설정 가능하며, 최종 스코어보드 공개는 관리자가 수행한다.
- 서비스 마스터/서비스 매니저/대회 운영자/대회 매니저는 이메일+비밀번호 로그인 구조다.
- 권한자 계정은 초대 또는 추가 후 메일을 통해 비밀번호를 설정한다.
- 권한자 계정은 비밀번호 재설정 및 비밀번호 분실 메일 재설정을 지원해야 한다.
- 참가 팀 로그인은 등록된 팀 로그인 식별자 기반 인증번호 로그인 구조다.
- 메인 페이지가 필요하다.
- 지원 언어는 C99, C++17, Python 3.13, Java 8 이다.
- 채점기는 백엔드와 분리한다.
- 공개 서비스 계층은 웹/API 서버로, 채점 계층은 별도 채점 서버로 분리한다.

- 채점은 FIFO 큐 기반으로 처리한다.
- 채점은 Docker 샌드박스에서 수행한다.
- 안정성과 로드 밸런싱은 최우선 요구사항이다.
- 대회 진행 중 데이터 추가 재채점은 허용하지 않는다.
- 대회 종료 후 재채점은 가능하지만, 대회 당시 스코어보드는 자동 변경하지 않는다.
