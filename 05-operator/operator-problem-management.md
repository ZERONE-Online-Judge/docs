# 대회 운영자 문제 추가 및 수정

문제 관리 페이지의 탭 구조와 상세 입력 항목은 [대회 운영자 문제 관리 페이지 상세](./operator-problem-page-details.md)를 기준으로 본다.

## 주요 작업

- 문제 추가
- 문제 수정
- 문제 삭제
- 문제 순서 변경
- 예제 입출력 관리
- 리소스 업로드/삭제
- 테스트케이스 수동 등록
- 테스트케이스 제너레이터 관리

## 문제 데이터

- 문제 코드
- 제목
- 본문
- 입력 설명
- 출력 설명
- 제약 조건
- 시간 제한
- 메모리 제한
- 표시 순서
- 채점 제도별 점수 설정

## 본문 작성 포맷

- 문제 본문과 입력/출력 설명, 제약 조건 등 글로 작성하는 영역은 서식 기능을 지원해야 한다.
- 최소한 `Markdown` 기반 서식 중 `h1`, `h2`, `h3`, `h4`, 기울임, 굵게, 표, 이미지 문법을 지원해야 한다.
- 제목 내부에서도 기울임과 굵게 강조 문법을 허용할 수 있어야 한다.
- 수식용 `LaTeX` 문법 입력을 지원해야 한다.
- 화면 렌더링은 `KaTeX` 등으로 일관되게 표시할 수 있어야 한다.
- 이미지는 업로드 후 `Markdown` 이미지 문법으로 참조할 수 있어야 한다.
- HTML 직접 입력은 허용하지 않아야 한다.
- 렌더링 과정에서 XSS를 방지할 수 있도록 입력 정제와 안전한 렌더링 규칙이 필요하다.

## 점수제 대회 추가 데이터

- 문제 만점 `100점`
- 일반문제 또는 서브태스크형 문제 구분
- 서브태스크 또는 기준 목록
- 기준별 테스트케이스 묶음
- 기준별 배점
- 배점 합계 검증 결과

## 문제 유형 판별 규칙

- 배점 구간이 `100점` 단일 하나뿐이면 일반문제로 본다.
- 최대치 `100점`을 포함하면서 `100점 미만`의 중간 배점 기준이 하나라도 있으면 서브태스크형 문제로 본다.
- 점수제 대회에서 일반문제는 `0점 또는 100점`만 나오는 문제로 처리한다.
- 서브태스크형 문제의 각 기준은 출제자가 지정한다.
- 각 기준은 해당 기준에 연결된 모든 테스트케이스가 정답 처리되어야 통과한 것으로 본다.
- 테스트케이스 진행 중 오답 처리가 발생하면 채점기는 이후 케이스를 진행하지 않고 즉시 해당 제출을 오답으로 확정한다.

## 리소스 정책

- 본문용 이미지 파일만 허용
- 최소 허용 확장자: png, jpg, jpeg, webp
- 파일당 최대 512MB
- 외부 URL 이미지는 허용하지 않고, 업로드된 내부 asset만 참조할 수 있다.

## 파일 저장 정책

- 문제 본문, 입력 설명, 출력 설명, 제약 조건은 DB에 저장한다.
- 예제 입력/출력은 DB에 저장한다.
- 문제 이미지와 첨부 리소스는 파일 스토리지에 저장하고 DB에는 메타데이터를 저장한다.
- 테스트케이스 입력/출력은 파일 스토리지에 저장하고 DB에는 버전, 파일 크기, checksum, 활성 상태 등 메타데이터를 저장한다.
- 제출 원본 코드와 채점 산출물은 제출 단위 디렉터리에 저장하고 DB에는 제출 메타데이터와 결과 상태를 저장한다.
- 파일의 실제 경로는 참가자나 공개 사용자에게 직접 노출하지 않는다.
- API는 권한 검사를 통과한 뒤 파일을 응답하거나 내부망에서만 유효한 짧은 접근 URL을 발급할 수 있다.

### 대회별 스토리지 구조

초기 구현은 MinIO 객체 스토리지를 기준으로 한다.
아래 구조는 MinIO object key prefix 기준이며, DB에는 논리 key와 메타데이터를 저장한다.

```text
/var/lib/zerone/
├── problem-resources/
│   └── contest-{contest_id}/
│       └── problem-{problem_id}/
│           └── {asset_id}
├── testcases/
│   └── contest-{contest_id}/
│       └── problem-{problem_id}/
│           └── version-{version}/
│               ├── input/
│               └── output/
└── submissions/
    └── contest-{contest_id}/
        └── submission-{submission_id}/
            ├── source
            ├── compile.log
            ├── run.log
            └── result.json
```

### 테스트케이스 버전 관리

- 테스트케이스는 기존 파일을 덮어쓰지 않고 새 version을 만들어 등록한다.
- DB의 active testcase set이 현재 채점에 사용할 version을 가리킨다.
- 새 테스트케이스 version은 검증이 끝난 뒤 active로 전환할 수 있다.
- 대회 진행 중에는 active testcase set 변경을 금지한다.
- active version 변경, 삭제, archive는 감사 로그 대상이다.
- 제출 채점 결과는 사용한 testcase set version을 추적할 수 있어야 한다.

## 문제 이미지 저장 정책

- 문제 본문 이미지는 사용자 파일명 기준으로 직접 조회하지 않아야 한다.
- 업로드된 이미지는 스토리지 내부에서 별도 `asset_id` 기반 이름으로 저장해야 한다.
- 원본 파일명은 표시용 메타데이터로만 보관하고, 실제 저장 파일명으로 사용하지 않아야 한다.
- 저장 경로는 운영 관리 편의를 위해 `contest/{contest_id}/problem/{problem_id}/images/...` 같은 prefix 구조를 가질 수 있어야 한다.
- 실제 파일 조회와 본문 참조는 파일명 검색이 아니라 `asset_id` 또는 이에 준하는 내부 식별자 기준으로 처리해야 한다.

### 메타데이터 관리

- 이미지마다 최소한 아래 메타데이터를 관리할 수 있어야 한다.
- `asset_id`
- `contest_id`
- `problem_id`
- `original_filename`
- `storage_key`
- `mime_type`
- `file_size`
- `sha256`
- `uploaded_by`
- `created_at`

### 본문 참조 규칙

- 문제 본문에서는 업로드된 이미지를 `Markdown` 이미지 문법으로 참조할 수 있어야 한다.
- 내부적으로는 실제 파일 경로 대신 `asset_id` 또는 내부 참조 식별자를 통해 렌더링할 수 있어야 한다.
- HTML `img` 태그 직접 입력은 허용하지 않아야 한다.

### 보안 및 확장성 원칙

- 확장자만 신뢰하지 말고 실제 파일 타입 검증이 가능해야 한다.
- 허용된 이미지 MIME만 통과시켜야 한다.
- 디렉터리 탐색이나 경로 추측으로 파일에 접근할 수 없어야 한다.
- 렌더링과 다운로드 경로는 접근 제어와 안전한 응답 헤더 정책을 가져야 한다.
- 스토리지 구현은 MinIO를 기준으로 하며, 이후 S3 호환 객체 스토리지로 이전 가능하게 유지한다.

## 운영 규칙

- 대회 시작 전에는 자유롭게 준비 가능
- 대회 진행 중에는 문제 본문과 데이터 수정 금지
- 운영자가 문제를 볼 수 있는 범위와 참가자가 볼 수 있는 범위는 분리해야 함
